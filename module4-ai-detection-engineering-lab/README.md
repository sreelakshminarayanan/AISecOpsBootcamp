# LLM Assisted Detection Engineering Lab

A local Detection as Code environment where a language model helps with detection research and drafting, and deterministic tooling decides what is actually valid, testable, deployable, and safe to promote. The lab runs Elasticsearch, Kibana, Sigma, pySigma, and Ollama behind a single Streamlit workbench.

The operating principle is simple and it is enforced in code, not just in the documentation:

> The model proposes. Deterministic checks validate. A human reviews before anything reaches production.

Nothing generated by the model is trusted by default. Every design and every Sigma rule passes through schema validation, field allowlists, ATT&CK allowlists, a Sigma syntax check, fixture tests against a real search backend, and a full dataset evaluation before it can be deployed.

---

## Overview

Detection engineering usually looks like this: read public research, draft a hypothesis, write a rule, test it against known good and known bad examples, deploy it, and then find out weeks later whether it actually holds up. This lab keeps that same shape but makes every step explicit and checkable.

The model is used at exactly two points in the workflow:

1. To turn analyst supplied research notes into a structured, testable research brief.
2. To turn an approved design into a Sigma rule proposal.

Everything else, including schema checks, field validation, ATT&CK tag validation, Sigma syntax checks, query conversion, regression testing, full dataset evaluation, deployment, alert verification, and rollback, is handled by ordinary Python code and established security tooling.

By the end of the lab you will have produced:

- A reviewed research brief
- A schema grounded detection design
- A portable Sigma rule
- A deterministic quality report
- A converted Elasticsearch query
- Positive and negative fixture test results
- A full dataset evaluation
- A deployed Kibana detection rule
- A verified Elastic Security alert
- A rollback record

The primary teaching scenario is encoded PowerShell launched outside approved management tooling. The malicious chain starts with Word spawning PowerShell with a hidden Base64 encoded command that decodes to Invoke WebRequest and Invoke Expression behavior, followed by an outbound connection, Certutil based retrieval, file creation, payload execution, registry persistence, a second encoded PowerShell execution, LSASS memory access, and outbound traffic. The rule has to tell that chain apart from Configuration Manager and Intune, which also run PowerShell with encoded commands as part of normal administration. A second scenario, WMIC based security tool tampering, is used as a capstone exercise where the learner repeats the full lifecycle without being handed the logic.

## Architecture

The lab runs as a Docker Compose project on a private bridge network named `detection-lab`.

| Service | Role | Host access |
|---------|------|-------------|
| `portal` | Streamlit Detection Engineering Workbench | http://localhost:8501 |
| `elasticsearch` | ECS telemetry, temporary fixture indexes, alert storage | http://localhost:9200 |
| `kibana` | Detection Engine API, rule operations, alert investigation | http://localhost:5601 |
| `ollama` | Local model inference | internal Docker network only |
| `ollama-init` | Downloads the configured model on first start | one time job |
| `elastic-setup` | Sets the Kibana system account password | one time job |
| `telemetry-init` | Generates and indexes telemetry, initializes the Detection Engine | one time job |
| `pipeline` | Persistent container for CLI and automation work | internal Docker network only |

Elasticsearch, Kibana, and the portal bind to `127.0.0.1`, so they are reachable only from the local machine. Ollama does not publish port `11434` to the host, which avoids clashing with an existing local Ollama install and keeps model access inside the lab network.

Two named volumes hold state across restarts: `elastic-data` for Elasticsearch indexes and `ollama-data` for downloaded models. `docker compose down` keeps both. `docker compose down -v` removes both, which means the dataset gets regenerated and the model gets downloaded again on the next start.

### Startup flow

```text
Elasticsearch starts as a secured single node cluster
elastic-setup sets the kibana_system password
Kibana starts and waits for Elasticsearch
telemetry-init runs scripts/bootstrap.py
  - generates the production simulation dataset
  - installs an index template for lab and fixture indexes
  - bulk indexes the dataset into lab-security-events-v1
  - initializes the Kibana Detection Engine alert index
pipeline and portal become available
ollama-init downloads the configured model in parallel
```

The workbench and every non model operation stay available while the model download is still running.

### Detection development flow

```text
Scenario YAML and analyst source notes
  -> LLM research brief (Stage 1)
  -> human review
  -> reviewed research + scenario + allowed ECS fields
  -> LLM detection design (Stage 2)
  -> JSON Schema validation
  -> human review
  -> reviewed design
  -> LLM Sigma proposal (Stage 3)
  -> deterministic quality gates (metadata, fields, UUID, ATT&CK tags, wildcards, sigma check)
  -> pySigma conversion to an ECS Lucene query
  -> fixture events loaded into a temporary Elasticsearch index
  -> query executed against fixtures (regression test)
  -> query executed against the full simulation dataset
  -> passing rule converted into a Kibana Detection Engine payload
  -> deployment through the Kibana API
  -> a positive fixture replayed with a fresh timestamp and event ID
  -> alert verification against the fresh replay
  -> previous payload stored for rollback on every update
```

### Dataset

The generator produces 252 ECS aligned Windows and Sysmon events: 241 benign or background events and 11 events that form the malicious encoded PowerShell chain, split across process, network, file, and registry categories. All network indicators use RFC reserved documentation ranges and inert test domains, so nothing in the dataset resolves or connects to anything real. The benign baseline uses a fixed random seed so event patterns and counts stay stable across runs, while timestamps are regenerated each time.

    theme.py                     Portal styling
  scripts/
    generate_telemetry.py         Deterministic dataset generator
    bootstrap.py                  Startup job: generate, index, initialize
  scenarios/                      Scenario YAML files that ground the model
  schemas/                        ECS fields, Sigma fields, design schema, ATT&CK catalog
  rules/                          validated (known good) and broken (intentionally invalid) rules

  labs/                           LAB_4_1 through LAB_4_8 write ups

  workspace/                      Generated research, design, and Sigma drafts

### `portal/app.py`

The Streamlit application and the main interface. It defines eight pages that map directly to the lab sequence: Range Overview, Detection Research, Detection Design, Sigma Workbench, Test and Tune, Telemetry Explorer, Deploy and Verify, and Capstone. The portal calls the same functions the CLI calls, writes generated artifacts to `workspace/`, `reports/`, and `deployments/`, and renders quality results, test results, and telemetry search directly in the browser.

### `detection_lab/config.py`

Centralizes service URLs, credentials, the configured Ollama model, the lab index name, and the project root, all overridable through environment variables. `LAB_ROOT=/workspace` and `PYTHONPATH=/workspace` keep every Python service, including the CLI, the portal, the bootstrap job, and the tests, on the same import path.

### `detection_lab/http_client.py`

A single JSON request wrapper used for every call to Elasticsearch, Kibana, and Ollama. It attaches authentication when the target is Elasticsearch or Kibana, enforces expected status codes, and converts transport failures into one consistent `ServiceError`.

### `detection_lab/llm.py`

Contains the three model assisted operations:

- `create_research_brief()`. Takes a scenario and analyst source notes, and asks the model to separate observed facts, analyst claims, and assumptions. The prompt explicitly tells the model to treat the source notes as untrusted evidence rather than instructions, and not to invent source content, telemetry, fields, or ATT&CK mappings.
- `create_detection_design()`. Takes the scenario, the reviewed research brief, and the allowed ECS field list, and asks the model to produce a structured design limited to that evidence and that field list.
- `draft_sigma_rule()`. Takes the reviewed design and asks the model for a portable Sigma rule using generic Windows process creation field names (`Image`, `CommandLine`, `ParentImage`, and so on), explicitly rejecting ECS field names inside the Sigma source.

Every call uses Ollama's chat endpoint with JSON output mode, a low temperature, and an 8192 token context window. Response parsing strips fenced code blocks and rejects anything that is not a valid JSON object.

### `detection_lab/quality.py`

Implements JSON Schema validation for the design artifact and an eleven gate quality check for the Sigma rule: YAML parsing, required metadata, a valid UUID, the expected Windows process creation log source, a portable field allowlist, an ATT&CK tag allowlist, presence of a detection condition, rejection of standalone wildcard values, a `sigma check` pass, and a minimum count of positive and negative fixtures. This module also contains `audit_content()`, which scans the repository for en dashes, em dashes, horizontal bar characters, and a small set of disallowed phrasing, and is what the CI workflow runs as a static gate.

### `detection_lab/sigma_tools.py`

Thin wrappers around two Sigma CLI subprocess calls:

```text
sigma check <rule>
sigma convert -t lucene -p ecs_windows <rule>
```

Conversion uses the Elasticsearch backend with the `ecs_windows` processing pipeline. The last non diagnostic line of the conversion output is extracted as the executable Lucene query.

### `detection_lab/elastic.py`

The operational core of the lab. It covers service health checks, Ollama model readiness, index templates, dataset statistics, telemetry search, NDJSON ingestion, bulk indexing, temporary fixture index creation and teardown, fixture and full dataset evaluation with precision and recall, fresh event replay, Kibana rule creation and update, deployment history, rollback, and alert verification. Fixture tests run against a temporary index named `lab-fixtures-<random value>` that is deleted once the test completes, so regression runs never touch the primary dataset.

### `detection_lab/evtx_import.py`

An optional path for importing authorized, sanitized Sysmon EVTX files and mapping process creation, network connection, file creation, and registry events into the same ECS shape used by the generated dataset.

### `detection_lab/cli.py` and `bin/lab`

The CLI exposes the full workflow for terminal use and CI: `status`, `research`, `design`, `draft`, `validate`, `convert`, `ingest`, `import-evtx`, `test`, `evaluate`, `replay`, `deploy`, `verify-alert`, `rollback`, `audit`, `pipeline`, and `acceptance`. `pipeline` runs the static gates, conversion, fixture tests, and full dataset evaluation in one pass. `acceptance` extends that path through deployment, replay, fresh alert verification, a second deployment revision, and rollback, and is the closest thing in the lab to an end to end proof that the lifecycle works.

### `scripts/generate_telemetry.py` and `scripts/bootstrap.py`

`generate_telemetry.py` builds the 252 event dataset with stable IDs, hashes, host and user identities, Sysmon metadata, scenario labels, and ground truth labels for what is malicious and what is expected to match. `bootstrap.py` is the container entry point that regenerates the dataset, indexes it, and initializes the Kibana Detection Engine index at startup.

### `scenarios/`, `schemas/`, `rules/`, `tests/`

`scenarios/` holds the YAML files that describe the behavior, evidence, and constraints the model is grounded on for each detection (currently encoded PowerShell and WMIC security tool tampering). `schemas/` holds the guardrails: the allowed ECS fields, the allowed portable Sigma fields, the JSON Schema for the design object, and an ATT&CK technique catalog that mapping tags are checked against. `rules/validated/` contains the known good encoded PowerShell rule, and `rules/broken/` contains an intentionally invalid rule used to demonstrate gate failures. `tests/fixtures/encoded_powershell/` holds two positive and four negative fixture events used for regression testing.

### `.github/workflows/detection-ci.yml`

Runs on changes to rules, tests, schemas, `detection_lab/`, `portal/`, `docker-compose.yml`, or `requirements.txt`. It installs dependencies, runs the content audit, validates the known good rule, converts it, and runs the pytest suite.

## Prerequisites

| Requirement | Detail |
|-------------|--------|
| Host | 64 bit machine capable of running Linux containers |
| Docker | Docker Engine with the Compose plugin on Linux, or Docker Desktop on Windows |
| Memory | At least 8 GB available to Docker, 12 GB is more comfortable |
| Disk | About 12 GB free |
| Network | Internet access on first start for container images and the Ollama model |
| Ports | `8501`, `5601`, and `9200` free on the host |
| Permissions | Ability to create Docker containers, networks, and volumes |

A local Python installation is not required. Python, Sigma CLI, pySigma, Streamlit, and every other dependency are installed inside the project image, so the only thing you install on the host is Docker.

The default model is `qwen2.5-coder:3b`, set in `.env.example` as `OLLAMA_MODEL`. Change it before the first `docker compose up` if you want a different model, since `ollama-init` pulls whatever is configured there.

## How to run on Windows

Install Docker Desktop, enable the WSL 2 based engine, and confirm it is running.

Open PowerShell and check the tooling:

```powershell
docker --version
docker compose version
```

From the project directory:

```powershell
cd .\module4-ai-detection-engineering-lab
Copy-Item .env.example .env
docker compose up --build -d
```

Check that services came up:

```powershell
docker compose ps
```

Check portal health:

```powershell
Invoke-RestMethod http://localhost:8501/_stcore/health
```

An `ok` response means the portal is up. Open http://localhost:8501 for the workbench and http://localhost:5601 for Kibana (username `elastic`, password `DetectionLab2026` unless you changed `.env`).

Track the model download, which runs in the background and does not block anything else:

```powershell
docker compose logs -f ollama-init
```

Stop the lab and keep data:

```powershell
docker compose down
```

Full reset, including the indexed dataset and the downloaded model:

```powershell
docker compose down -v
```

## Using the CLI directly

The `pipeline` container stays running and can be used for command line work or CI, either through `docker compose exec` or `make`:

```bash
docker compose exec pipeline /workspace/bin/lab status

docker compose exec pipeline /workspace/bin/lab research \
  --scenario scenarios/encoded_powershell.yml \
  --notes workspace/research_notes.txt \
  --output workspace/encoded_powershell_research.json

docker compose exec pipeline /workspace/bin/lab design \
  --scenario scenarios/encoded_powershell.yml \
  --research workspace/encoded_powershell_research.json \
  --output workspace/encoded_powershell_design.json

docker compose exec pipeline /workspace/bin/lab draft \
  --design workspace/encoded_powershell_design.json \
  --output workspace/encoded_powershell.yml

docker compose exec pipeline /workspace/bin/lab validate \
  --rule workspace/encoded_powershell.yml \
  --fixtures tests/fixtures/encoded_powershell

docker compose exec pipeline /workspace/bin/lab test \
  --rule workspace/encoded_powershell.yml \
  --fixtures tests/fixtures/encoded_powershell

docker compose exec pipeline /workspace/bin/lab evaluate \
  --rule workspace/encoded_powershell.yml \
  --index lab-security-events-v1
```

Makefile shortcuts for the same container:

```bash
make up        # docker compose up --build -d
make down      # docker compose down
make status    # lab status inside the pipeline container
make logs      # follow all service logs
make reset     # docker compose down -v
make shell     # open a shell inside the pipeline container
make pipeline  # run validate, convert, and fixture tests on the validated rule
make audit     # run the content audit
```

The full acceptance path, covering validation, conversion, fixture and dataset tests, deployment, replay, alert verification, a revision, and rollback, is run with:

```bash
docker compose exec pipeline /workspace/bin/lab acceptance --timeout 240
```

A passing run ends with `{"passed": true}` and writes `reports/acceptance_report.json`.

## Tests

The Python test suite runs outside Docker if you have the dependencies installed, or inside the pipeline container:

```bash
docker compose exec pipeline pytest -q
```

It covers EVTX mapping, telemetry generation, the quality gates, and the portal and Compose configuration. The GitHub Actions workflow runs the same audit, validation, conversion, and pytest steps on every relevant change.

## Troubleshooting

**Portal reports no module named detection_lab.** Rebuild the Python images so `PYTHONPATH=/workspace`, `LAB_ROOT=/workspace`, and the `/workspace` working directory take effect:

```bash
docker compose down --remove-orphans
docker compose build --no-cache telemetry-init pipeline portal
docker compose up -d
```

**Portal does not open.** Check its status and logs, and if port 8501 is taken, change `PORTAL_PORT` in `.env` and recreate it:

```bash
docker compose ps portal
docker compose logs --tail=150 portal
docker compose up -d --force-recreate portal
```

**Pipeline is not running.** It waits on telemetry initialization, so check that job's logs:

```bash
docker compose logs --tail=150 telemetry-init
docker compose logs --tail=150 elastic-setup
```

**Model stays in initializing.** Follow the download and, if needed, restart Ollama:

```bash
docker compose logs -f ollama-init
docker compose restart ollama
docker compose up -d ollama-init
```

**Start over completely.** This deletes indexed data and the downloaded model, both of which get rebuilt automatically on the next start:

```bash
docker compose down -v
docker compose up --build -d
```

## Scope and limits

This is a learning environment, not a production deployment. It runs a single node Elasticsearch cluster, uses default local credentials, serves HTTP without TLS, has no enterprise identity provider or multi tenant access control, uses synthetic telemetry instead of live connectors, keeps a small fixture set, stores rollback history in a local file, and has no approval workflow, high availability, or automated feedback from alert outcomes. These gaps are intentional for a local lab and are the actual engineering work required to take this lifecycle pattern into a production environment. The dedicated lab cookbook covers what that production path looks like in more detail, including a production reference architecture, a detection design contract, telemetry contracts, a test hierarchy, and a phased rollout plan.
